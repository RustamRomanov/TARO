# Техническая реализация блока «Таро» (полное описание)

**Актуализация структуры проекта:** апрель 2026. Актуальные маршруты API: OpenAPI `/docs` и [BACKEND.md](./BACKEND.md).

## 1) Назначение и границы модуля

Блок «Таро» в приложении реализует полный пользовательский сценарий:

1. Выбор расклада и колоды.
2. Проверка формулировки вопроса.
3. Выбор карт пользователем в интерфейсе.
4. Получение трактовок по каждой карте через AI.
5. Получение общего итога расклада через AI.
6. Сохранение истории запроса на backend.
7. Отправка итога пользователю в личные сообщения Telegram-бота (опционально).

Модуль состоит из:

- frontend-страницы с многофазным UX и анимациями (`frontend/src/pages/Tarot.jsx`);
- backend API-эндпоинтов Таро (`app/api/routes.py`);
- сервисов лимитов и доступа (`app/services/limits.py`);
- AI-клиента с fallback-логикой (`app/services/ai_client.py`);
- проверки Telegram `init_data` (`app/core/security.py`);
- моделей БД для лимитов и истории (`app/db/models/user.py`, `app/db/models/history.py`);
- подготовленных визуальных описаний карт (`app/data/tarot_card_descriptions.json`).

## 2) Состав backend и контракты API

Все основные эндпоинты Таро находятся в `app/api/routes.py`.

### 2.1 `POST /api/tarot/validate-question`

### Назначение
Проверка, что вопрос пользователя осмысленный и подходит под выбранный расклад.

### Вход

```json
{
  "question": "Что меня ждет в карьере в ближайшие месяцы?",
  "spread": "basic"
}
```

### Выход

```json
{
  "valid": true,
  "message": null
}
```

или:

```json
{
  "valid": false,
  "message": "Пожалуйста, переформулируйте вопрос более понятно."
}
```

### Детали реализации

- Сначала локально проверяется «бессмыслица» (`_looks_like_gibberish`):
  - очень короткий текст;
  - набор букв без пробелов/знаков.
- Если локальная эвристика не сработала, выполняется AI-проверка с требованием вернуть строгий JSON.
- Если AI не вернул корректный JSON или упал, модуль **не блокирует** пользователя: возвращается `valid=true`.

---

### 2.2 `POST /api/tarot/draw`

### Назначение
Получение трактовки для одной карты или для «общего итога расклада».

### Вход (карта)

```json
{
  "deck": "hermetic",
  "question": "Как мне действовать в отношениях?",
  "card_name": "Влюбленные",
  "card_id": "Major06",
  "position": 2,
  "total": 6,
  "position_name": "Чувства",
  "init_data": "<telegram webapp init_data>"
}
```

### Вход (итог расклада)

```json
{
  "deck": "hermetic",
  "question": "Общий итог расклада. Вопрос: ... Карты: ...",
  "card_name": "Общий расклад",
  "init_data": "<telegram webapp init_data>"
}
```

### Выход

```json
{
  "card_name": "Влюбленные",
  "interpretation": "..."
}
```

### Детали реализации

1. Проверка Telegram-пользователя из `init_data`.
2. Проверка лимита через `check_limits(..., "tarot")`.
3. Формирование промпта с контекстом:
   - вопрос;
   - позиция карты в раскладе;
   - визуальное описание карты (если есть `card_id`);
   - гендерный контекст (см. раздел 6).
4. Вызов AI.
5. При успехе:
   - инкремент дневного счетчика Таро (`increment_daily`),
   - запись в `history` (`type=tarot`, `request_content`, `response_content`).
6. При ошибке AI — fallback-текст: `Звезды молчат. Попробуй позже.`

---

### 2.3 `POST /api/tarot/share`

### Назначение
Отправка результата расклада в личный чат с ботом.

### Вход

```json
{
  "init_data": "<telegram webapp init_data>",
  "question": "Мой вопрос",
  "overall": "Итоговая трактовка",
  "cards": [
    {
      "position_name": "Ситуация",
      "name": "Императрица",
      "meaning": "Краткий текст",
      "image": "https://..."
    }
  ]
}
```

### Выход

```json
{
  "ok": true,
  "message": "Расклад отправлен..."
}
```

### Детали реализации

- Проверяется наличие `TELEGRAM_BOT_TOKEN`.
- Из `init_data` извлекается `user_id`.
- Бот отправляет:
  1) текстовый блок с вопросом/картами/итогом,
  2) до 10 изображений карт (если в `cards` переданы HTTP URL).

## 4) Бизнес-логика backend

### 4.1 Генерация трактовки карты

В `tarot_draw` используется два режима:

- **Режим карты**: `card_name != "общий расклад"`
  - 4-6 предложений;
  - акцент на взаимосвязь изображения карты и вопроса.
- **Режим итога**: `card_name == "общий расклад"`
  - 5-7 предложений;
  - синтез всех карт в единый вывод и совет.

### 4.2 Визуальный контекст карты

- По `card_id` ищется описание в `app/data/tarot_card_descriptions.json`.
- Если описание найдено, добавляется в prompt как «что изображено на карте».
- Это позволяет делать трактовку не только по названию аркана, но и по визуальной сцене.

### 4.3 Гендерный контекст

Функция `_get_user_gender_hint(db, telegram_id, request_text)`:

1. Ищет первичный профиль пользователя (`Profile.is_primary == True`).
2. Если пола нет:
   - пытается вывести по имени (`infer_gender_from_name`);
   - если не удалось и вопрос достаточно длинный, пытается вывести AI-методом (`infer_gender_from_text_by_ai`).
3. Возвращает подготовленную текстовую подсказку для системного промпта (`gender_hint_for_prompt`).

### 4.4 История

После успешной генерации в таблицу `history` записывается:

- `user_id`,
- `type = tarot`,
- `request_content` (обрезается до 2000 символов),
- `response_content` (обрезается до 5000 символов).

## 5) Лимиты и доступ

Логика в `app/services/limits.py`.

### 5.1 Политика

- Для free: `DAILY_LIMIT_TAROT_FREE = 1`.
- Для trial/full_access: лимит не ограничивается этим счетчиком.

### 5.2 Проверка

`check_limits(db, telegram_id, "tarot")`:

1. Создает пользователя при первом запросе (`_ensure_user`), если его еще нет.
2. Проверяет необходимость суточного сброса (`_reset_daily_if_needed`) по UTC дате.
3. Если пользователь free и `daily_tarot >= 1` -> `HTTPException(403, "Лимит исчерпан...")`.

### 5.3 Инкремент

`increment_daily(db, telegram_id, "tarot")` повышает `daily_tarot` на 1 через SQL update.

## 6) Авторизация Telegram (init_data)

Реализация в `app/core/security.py`.

### Механизм

1. Парсится query-string `init_data`.
2. Извлекается `hash`.
3. Формируется `data_check_string` из остальных параметров (отсортированных).
4. Вычисляется HMAC-SHA256 по схеме Telegram Mini App:
   - secret key = HMAC("WebAppData", TELEGRAM_BOT_TOKEN),
   - compare digest с `hash`.
5. Если подпись валидна, из `user` JSON извлекается `id`.

### Важное следствие

Если `TELEGRAM_BOT_TOKEN` не задан, проверка не пройдет, и Таро-эндпоинты, требующие `init_data`, будут возвращать отказ.

## 7) AI-интеграция

AI-клиент: `app/services/ai_client.py`.

### 7.1 Провайдеры

- Primary:
  - OpenAI (`OPENAI_API_KEY`) — приоритетный режим;
  - либо `AI_*` провайдер (если OpenAI key отсутствует).
- Text fallback:
  - DeepSeek (`DEEPSEEK_API_KEY`).
- Vision fallback:
  - GPT через `OPENAI_GPT_API_KEY` (для vision-методов).

### 7.2 Для Таро используется текстовая генерация

Метод `generate_text()`:

1. Пробует primary модель.
2. При ошибке пробует DeepSeek.
3. Если оба недоступны — исключение.

### 7.3 Параметры генерации Таро

- `validate-question`: `max_tokens=180`.
- Карта: `max_tokens=320`.
- Итог расклада: `max_tokens=400`.

## 9) Frontend: архитектура и state-machine

Главная страница: `frontend/src/pages/Tarot.jsx`.

### 9.1 Ключевые состояния экрана

- `select` — выбор расклада/колоды + ввод вопроса.
- `animating` — анимации подготовки колоды, раскрытие веера, выбор карт.
- `result` — просмотр карт поочередно + итог расклада.

Подфазы анимации:

- `deck`
- `shuffling`
- `spreading`
- `picking`
- `deckExiting`
- `uniteFlip`

### 9.2 Ключевые состояния данных

- `selectedSpreadId`
- `activeDeckId`
- `question`
- `shuffledDeckForPick`
- `pickedIndices`
- `spread` (массив карт с `meaning`, `image`, `loading`)
- `overallInterpretation`
- `isDrawing`
- `spreadFailed`

### 9.3 Источники данных на фронте

- `frontend/src/data/tarotSpreads.js`:
  - конфигурации раскладов (basic, relationship, celtic),
  - позиции карт,
  - тексты подсказок,
  - рекомендованные колоды.
- `frontend/src/data/tarotDecks.js`:
  - динамическая загрузка изображений карт из `assets/taro`,
  - выделение рубашки (`рубаш`) и изображения колоды (`колода`),
  - формирование списка карт.
- `frontend/src/data/tarotCardNamesRu.js`:
  - локализация названий карт.

### 9.4 UX-поток запуска расклада

1. Пользователь выбирает расклад, колоду, вопрос.
2. Нажимает «Сделать расклад».
3. Фронт вызывает `validate-question`.
4. При `valid=false` показывает ошибку рядом с вопросом и не стартует анимацию.
5. При `valid=true` запускаются анимации и выбор карт.
6. По мере выбора каждой карты отправляется `POST /api/tarot/draw`.
7. Когда выбраны все карты, отправляется отдельный `draw` с `card_name="Общий расклад"`.
8. В `result` показываются трактовки и итог.

### 9.5 Retry и сетевая устойчивость на фронте

Для трактовки конкретной карты (`drawSingleCardForPosition`):

- таймаут запроса 15 секунд через `AbortController`;
- до 4 повторов;
- проверка «пустой/молчаливой» трактовки;
- кнопка «Повторить» на карточке результата.

### 9.6 Голосовой ввод вопроса

Используется Web Speech API:

- `window.SpeechRecognition` / `webkitSpeechRecognition`;
- `ru-RU`, `interimResults=true`, `continuous=true`;
- на неподдерживаемых браузерах — мягкая деградация с сообщением пользователю.

## 10) Текущая реализация share на frontend

Кнопка «Поделиться» в `FinalBlock` вызывает:

- `POST /api/tarot/share`
- передает `init_data`, `question`, `overall`.

В текущем коде `cards` отправляется как пустой массив `[]` в этом конкретном вызове.  
Backend поддерживает отправку списка карт, но frontend сейчас не передает заполненный массив карт в этом месте.

Это не ломает отправку (итог уходит), но ограничивает полноту сообщения (без карточек/изображений карт в payload фронта).

## 11) Обработка ошибок (backend + frontend)

### 11.1 Backend

- `401` в `draw`:
  - если нельзя получить пользователя из `init_data`.
- `403`:
  - если исчерпан лимит free-пользователя.
- AI-ошибки:
  - fallback-текст вместо падения endpoint.
- `tarot/share`:
  - возвращает `ok=false`, `message=...` при проблеме токена бота, auth, Telegram API.

### 11.2 Frontend

- На `403` показывает текст лимита из `detail`.
- На сетевые ошибки показывает fallback-тексты (`Звезды молчат...`, `Не удалось загрузить...`).
- Для карточек доступен ручной retry.

## 12) Производительность и эксплуатационные особенности

### 12.1 Почему модуль «тяжелый» по запросам

Каждая карта в раскладе — отдельный backend AI-запрос:

- basic: 3 карты + 1 итог = 4 запроса,
- relationship: 6 + 1 = 7 запросов,
- celtic: 10 + 1 = 11 запросов.

При слабом соединении это заметно в UX (поэтому в UI есть поэтапная отрисовка и retries).

### 12.2 Предзагрузка медиа

`useTarotDecks` делает батчевую предзагрузку изображений карт, чтобы снизить «рывки» во время анимаций и показа результата.

## 13) Конфигурация окружения (критичное для Таро)

Минимально важные переменные:

- `TELEGRAM_BOT_TOKEN` — обязателен для корректной валидации `init_data` и для share.
- `OPENAI_API_KEY` (или альтернативный `AI_API_KEY`) — основной text AI.
- `AI_TEXT_MODEL` — модель текста (если не задано, берется дефолт).
- `DEEPSEEK_API_KEY` — text fallback.

Если `TELEGRAM_BOT_TOKEN` отсутствует, Таро-авторизация через `init_data` не пройдет.

## 15) Краткая сквозная последовательность (end-to-end)

1. Пользователь открывает экран Таро в Telegram WebApp.
2. Выбирает расклад/колоду, вводит вопрос.
3. Фронт валидирует вопрос (`/api/tarot/validate-question`).
4. Если расклад `single`: фронт сразу запускает `POST /api/tarot/draw-batch` без веера.
5. Если расклад multi-card: пользователь выбирает карты из веера, затем фронт вызывает `POST /api/tarot/draw-batch` один раз на весь расклад.
6. Backend (`/api/tarot/draw-batch`):
   - валидирует `init_data`,
   - проверяет лимит,
   - назначает reversed-state (если включено),
   - формирует единый batch prompt,
   - вызывает AI,
   - сохраняет `TarotReading`,
   - увеличивает счетчик `daily_tarot`.
7. Фронт показывает интерпретации карт и либо сразу итог (простые расклады), либо сначала диалог (глубокие расклады 6/10) с кнопкой «Получить итог расклада».
8. Для 6/10 карт пользователь может продолжить консультацию через `POST /api/tarot/chat`.
9. По кнопке «Поделиться» фронт вызывает `/api/tarot/share`; backend отправляет результат в Telegram-бот.
10. Если текст слишком длинный, `/api/tarot/share` автоматически дробит его на части (`Часть i/N`) в пределах лимита Telegram.

## 16) Итог по текущему качеству реализации

Сильные стороны:

- batch-архитектура для раскладов: один запрос на расклад вместо N запросов по картам;
- полноценный диалоговый режим для глубоких раскладов;
- reversed-карты с визуалом и учетом в prompt;
- интеграция с глобальным профилем (`useProfile`) без локального селектора;
- история/статистика/recurring-карты с фильтрацией по профилю;
- cinematic UX: усиленные анимации shuffle/spread/pick/deck-exit, full-screen Vanta/fog;
- отдельный deep-flow для `10 карт` (режимы консультации и просмотра карт);
- отдельный расклад `financial` с зеленой палитрой, практическим шагом на 7 дней и чеклистом с localStorage;
- Telegram share отправляет полный текст без обрезки, включая авто-чанкование длинных сообщений.

Текущие ограничения:

- для legacy fallback (`/api/tarot/draw`) финальный итог строится старым способом и менее структурирован;
- при недоступности Telegram Bot API шаринг остается недоступным (возвращается контролируемая ошибка);
- полный UX-тест deep-flow лучше проверять на реальном Telegram-клиенте (desktop/mobile отличаются по safe-area/keyboard).

1. Добавлен расклад `financial` (5 карт) на frontend и backend.
2. Для `financial` добавлена отдельная цветовая тема фона (зеленая палитра Vanta).
3. Для `financial` введен отдельный блок практического совета на 7 дней + чеклист прогресса.
4. Чеклист `financial` сохраняется в `localStorage` и восстанавливается при повторном открытии.
5. Для `10 карт` внедрен отдельный консультационный режим (переключение `Консультация / Карты`).
6. Для глубоких раскладов итог показывается по действию пользователя (`Получить итог расклада`) или после фактического диалога.
7. Улучшен UX загрузки карты: анимация звездочек, левитация, показ рубашки во время чтения.
8. Исправлен reversed-индикатор: `Карта выпала перевёрнутой`.
9. Исправлены модальные окна истории/статистики: корректная позиция, safe-area и скролл.
10. Исправлен Telegram share:
    - снято обрезание текста карточек и итогового блока;
    - добавлено автоматическое разбиение длинного текста на части;
    - добавлена нумерация частей (`Часть i/N`).
